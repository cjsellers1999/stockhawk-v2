# Prototype the residential-IP crawl scheduler

## Question

What adaptive scheduling model produces the fastest safe combined Catalog Discovery and Stock Monitoring on one Mac mini and residential IP? Build a throwaway logic prototype using representative Connector costs to test global and per-Storefront concurrency, priority queues, caching, conditional work, jitter, retry/backoff behavior, failure isolation, full-snapshot reconciliation, and future alert latency; use the results to choose measurable pacing and freshness policies.

## Comments

- Use Connector Run Metrics from both HTTP and granted browser work. The prototype must measure browser cost separately and choose safe recurring cadence and priority for Browser Access Grants rather than treating browser fallback like ordinary requests.
- Prototype asset: [Residential-IP scheduler prototype verdict](../prototypes/residential-ip-scheduler/NOTES.md), preserving the accepted answer after removal of the throwaway executable shell.
- Initial deterministic sweeps favor the Adaptive fair control shape without claiming real-world capacity: across representative scenarios it kept modeled failures near the cautious serial policy while Fixed greedy produced roughly 17–62% failed network work. The Seed List sample showed that a saturated adaptive controller can drain much more catalog work than serial scheduling, but all sample figures test queue behavior only and are not rollout-duration forecasts.
- The prototype preserves the already-accepted invariants: one active job per Storefront, resumable catalog quanta, immediately useful Partial observations, certification only after complete work, failure isolation, separately measured and adaptively expanded browser access, coalesced overlapping stock work, and zero disappearance reconciliation from partial runs.
- **Owner direction — Network-Limited Scheduling:** Adaptive fair is the accepted policy family, but no production concurrency, browser, CPU, RAM, or other hardware quota may become its throughput ceiling. It must maximize successfully committed useful observations per wall-clock time, ramping global and per-Storefront concurrency through already-due work until source-specific throttling/challenges or correlated residential-IP pressure reveal the current network-safe limit. A numerical simulator guard is not a production policy, and raw attempted-request rate is not the objective when retries reduce useful throughput.
- **Owner hypothesis — isolated browser-context pool:** Compare one governed browser lane with an adaptive pool of ordinary isolated browser contexts. Every context must still acquire permits from the same residential-IP Crawl Request Broker and obey the same per-Storefront controls. The experiment may test session isolation, context reuse, and legitimate parallel rendering; it must not vary fingerprints or viewport traits to masquerade as separate users, interact with CAPTCHAs, or evade anti-bot controls. A challenge is typed pressure evidence that stops and backs off the affected work.
- **Prototype result — conditional browser-pool win:** In a deliberately saturated one-hour browser discovery wave, the adaptive isolated-context pool completed 3,457 useful requests versus 1,309 through one browser lane (about 2.6×), briefly probed seven contexts, crossed the synthetic safe point of five, then backed down to three. It did not improve an ordinary dispersed Browser-heavy run where the browser lane was not the bottleneck. The production implication is to expand ordinary isolated contexts only behind due work and successful-throughput evidence, while every context consumes the same shared-IP and per-Storefront permits.

## Answer

StockHawk V1 uses an **HTTP-first hybrid Network-Limited Scheduler**. A Connector Adapter selects the lightest reliable public mechanism for each Storefront: governed HTTP remains the normal fast path, while a Browser Access Grant permits browser work only on audited routes where HTTP cannot supply dependable catalog or stock evidence. The scheduler never converts HTTP-capable work into browser work merely because multiple contexts are available.

All outbound work passes through one Crawl Request Broker. It maintains deadline-aware fair queues for Catalog Discovery, official Master Catalog refresh, and Stock Monitoring; admits work only when it is due; and optimizes successfully committed useful observations per wall-clock time rather than raw request attempts. Stock freshness urgency, catalog lateness, first-answer value, and future watched-item urgency affect priority, while bounded catalog quanta guarantee that long discoveries yield without losing their checkpoint or searchable Partial observations. Neither workload may starve.

Concurrency is learned independently at three scopes: the shared residential IP, each Storefront, and browser-required execution. Initial Storefront Integration constraints provide the safe starting point. Saturated clean work across independent Storefronts raises capacity quickly before the first pressure signal and cautiously afterward; Retry-After, repeated throttles, latency degradation, challenges, and correlated failures reduce the relevant scope with jittered exponential backoff. A Storefront-specific failure slows and isolates only that Storefront. Correlated failures across otherwise independent Storefronts reduce shared-IP capacity. Capacity grows only behind real due work—StockHawk never creates measurement-only probe traffic—and CPU, RAM, or an arbitrary fixed production concurrency number must not intentionally become V1's ceiling.

Browser-required work starts with one ordinary isolated context and may expand into an adaptive context pool when a real browser backlog saturates the lane and clean completions improve useful throughput. Contexts isolate Storefront cookies and storage but remain one network identity: every page and request consumes the same broker permits and per-Storefront limits. One browser challenge stops and backs off the affected Storefront; only correlated challenge evidence contracts the shared pool. Fingerprint or viewport impersonation, CAPTCHA interaction or bypass, proxies, and anti-bot evasion are not part of the design.

The broker and Connector cooperate to avoid work: memory and durable caching where semantics allow, conditional requests, jittered due times, Storefront-local batching, and coalescing when one discovery response also gives trustworthy stock evidence. Only one Connector job owns a Storefront at a time. Catalog work commits bounded progress immediately, but interruption or failure never certifies completeness and never reconciles an unseen listing as disappeared. A complete certified snapshot alone may perform absence reconciliation; failed stock work preserves the prior Stock Status and observation time.

Connector Run Metrics from ordinary work are the feedback surface: HTTP and browser operations, request-equivalent cost, bytes, cache/conditional outcomes, active versus broker-wait time, useful observations committed, latency, retries, throttles, challenges, and server guidance. Scheduler and Health verification track successful observations per wall-clock time, failure/retry rate, queue depth and age, p95 stock and catalog lateness, current observation/certification age, first-Partial and full-certification latency, and future change-detection latency. The prototype's hidden capacities are not production defaults; the real Mac mini and residential-IP limits are learned continuously from allowed Connector work.

The accepted synthetic comparison supports the control shape, not a universal speed claim. During a deliberately saturated browser-only wave, an adaptive isolated-context pool completed about 2.6 times as much useful work as one browser lane before pressure caused it to contract. During a normal dispersed workload, extra contexts added no throughput because browser execution was not the bottleneck. Therefore the pool is an adaptive tool for browser-required backlog, while governed HTTP remains preferred everywhere it is reliable.
