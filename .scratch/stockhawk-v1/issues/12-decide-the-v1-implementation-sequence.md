# Decide the V1 implementation sequence and handoff

Type: grilling
Label: wayfinder:grilling
Status: resolved
Triage: ready-for-human
Blocked by: 11

## Question

Given every resolved architecture, audit, interaction, operations, and verification decision, what tracer-bullet implementation sequence reaches useful end-to-end Jellycat results early without weakening the final completion contract? Decide vertical slices, dependency order, proof gates, the point at which the full Storefront rollout begins, and the exact artifacts to hand to execution.

## Upstream handoff constraints

- Build the three-plane verification system alongside the first vertical slice: deterministic fixtures/tests, the actual-Mac representative release rehearsal, and live per-Storefront qualification.
- No fixture or shared Platform Adapter result may substitute for per-Storefront Catalog Certification and shopper-visible Stock Semantics Validation.
- Full rollout starts only after the deterministic suite and representative Mac gate pass; first-time recurring scheduling remains behind each Integration's certification, stock-semantics, and target-accounting gate.
- The first representative-load slice must exercise at least 100,000 Offers, prove accepted search latency and scheduler degradation behavior, and measure database/detail/media growth so cache quota and disk high-water configuration are set before rollout.
- Every slice must extend the commit-keyed Evidence Bundle; rollout continuously regenerates the deterministic full-input closeout reconciliation, and Partial can never be hidden as complete.

## Comments

- **Decision — begin with one real end-to-end Storefront:** The first implementation milestone takes one real Storefront through immutable Seed Site provenance, resumable onboarding, official Master Catalog refresh, complete catalog and variant discovery, catalog matching, shopper-validated Stock Monitoring, PostgreSQL persistence and Change Events, Search, Health, and verification evidence. It uses the final module seams and operational topology rather than throwaway code, fake-only data, or disconnected horizontal foundations. Later Storefronts and Connector shapes widen this working path.
- **Decision — capability proof gates the full rollout:** After the first Storefront, a small real pilot must prove the shared HTTP Adapter on two distinct Storefront Integrations, another common platform shape, one audited browser-required Integration, one Bespoke Connector Adapter, and live Dead, Non-Store, and certified zero-Jellycat Dormant outcomes. The actual Mac must also pass the deterministic suite and 100,000-Offer representative release rehearsal. This capability matrix—not an arbitrary pilot-store count—opens the complete Seed List rollout; useful pilot results remain searchable before then.
- **Decision — rollout follows Connector leverage, not spreadsheet order:** After the pilot gate, onboarding coalesces proven duplicate/alias Candidates onto one Storefront and prioritizes the Connector capability that unlocks the largest remaining Candidate cohort. Deterministic ordering within a cohort keeps work and reports reproducible. A separately bounded fair lane continuously advances unknown-platform and likely Bespoke cases so maximum early coverage cannot starve rare one-off Storefronts. The immutable workbook reconciliation—not queue order—proves every Candidate was eventually handled.
- **Decision — useful operation begins before final V1 acceptance:** After the capability pilot and actual-Mac release gates pass, the owner may use StockHawk daily while full onboarding continues. Every individually published Integration runs its normal proof-gated schedule; confirmed and explicitly Partial observations stay searchable; Search and Health disclose rollout coverage and incompleteness. This early operational state is never called completed V1: final acceptance still waits for every Candidate Site and Storefront branch to reach an accepted terminal outcome and for the generated closeout ledgers to reconcile.
- **Decision — deepen the Connector seam from real evidence:** Implementation begins with the accepted two-job Connector contract, common conformance kit, broker boundary, and only the capabilities required by the first real Adapter. The second Storefront proves configuration reuse; the next common platform, browser-required, and Bespoke pilots extend shared machinery only for observed reusable needs. No speculative universal framework, retailer-name conditional inside a shared Adapter, or lowest-common-denominator abstraction delays the first complete path.
- **Decision — pressure-test immediately after the first complete path:** As soon as the first real Storefront works end to end, load the representative 100,000-Offer dataset and run database, Change Event, Search Document, query-plan, UI, scheduler, retention, and storage-growth verification before widening Connector coverage. The data generator preserves observed first-Storefront distributions while adding all required status, match, failure, history, and Storefront shapes. Performance fixes therefore reshape one path rather than forcing late rewrites across many Adapters. The later actual-Mac release rehearsal reruns this gate in the final topology; an early development-machine pass cannot replace it.
- **Decision — expand rollout through widening evidence gates:** After the pilot opens rollout, process initial waves of 25, 100, and 500 previously unhandled Candidate Sites before releasing the remainder. Connector-leverage and fair-lane ordering still select each wave. Between waves, automatically reconcile inputs/branches, review unexpected alias and duplicate rates, inspect Partial and typed-failure distributions, validate classification and certification anomalies, measure disk/media growth and residential-IP behavior, and rerun affected deterministic contracts. A systemic defect or unexplained broad anomaly stops wave expansion for repair and replay; an isolated difficult Storefront suspends only its own case. These are agent-owned gates without routine owner approval.
- **Decision — select the first Storefront by evidence, not by name:** A bounded implementation-time preflight chooses a normal retailer from the cleaned Seed List on a common high-leverage platform, with a credible complete public HTTP catalog surface, exact variants, images, and contemporaneously observable in-stock and out-of-stock examples, without CAPTCHA or exceptional access. The exact selection and rejected alternatives are recorded in the Evidence Bundle. The official Jellycat Storefront is not the retailer canary; official Master Catalog refresh is a separate input, and retailer-only discoveries may still enter the accepted open-world matching path.
- **Decision — execution receives four local handoff artifacts:** Closing the map produces one concise implementation Spec that links rather than forks canonical domain decisions, one dependency-ordered tracer-bullet ticket set, one verification matrix mapping every promise to deterministic, actual-Mac, and live-Storefront evidence, and one rollout runbook covering first-Storefront selection, the capability pilot, widening waves, repair/resume, and final reconciliation. These remain local Markdown execution inputs; implementation begins afterward through the ticket frontier rather than being mixed into the planning map.

## Answer

V1 implementation follows a proof-carrying tracer route rather than horizontal subsystem construction. First establish the production-shaped runtime and deterministic gates, then commit and search one synthetic Offer through the real Persistence Boundary and secure one optimistic owner command. Import immutable Seed provenance, build resumable Onboarding Cases, establish the brokered two-job Connector contract, and bootstrap the official open-world Master Catalog before selecting the first retailer.

The first real Storefront is chosen by bounded evidence from the cleaned Seed List: a normal high-leverage shared platform with complete public HTTP enumeration, exact variants, images, and observable in-stock/out-of-stock states. Take it through browser-assisted audit, Integration publication, bounded discovery, immediate Partial search value, central Catalog Certification, shopper-visible Stock Semantics Validation, full target accounting, recurring scheduling, Search, Health, and Change Events using final seams rather than throwaway code.

Immediately pressure-test that complete path with at least 100,000 representative Offers. Correct query, concurrency, scheduler, retention, event, UI, and storage behavior and derive media-cache/disk thresholds before Adapter expansion. Then run a capability pilot proving shared HTTP reuse on a second Storefront, another common platform, one audited browser-required Integration, one Bespoke Connector Adapter, and live Dead, Non-Store, and certified zero-Jellycat Dormant outcomes. Every Integration supplies its own catalog and shopper-visible stock proof.

Full rollout opens only after the pilot, deterministic suite, and actual-Mac release rehearsal pass. At that point the owner may use qualified and explicitly Partial data daily while Health shows incomplete rollout coverage; useful operation is not mislabeled completed V1. Rollout prioritizes Connector leverage while preserving a bounded fair lane for unknown and one-off Storefronts, expands through cumulative 25, 100, and 500 Candidate gates, then drains the remainder. Systemic anomalies stop expansion for repair; isolated cases suspend locally and resume from committed evidence/checkpoints.

Execution receives the four linked local artifacts. The approved ticket graph keeps each fixed implementation slice independently demonstrable, then generates deterministic one-context sub-batch tickets from live rollout evidence instead of pretending all roughly 2,247 Candidates fit one static task. V1 finishes only when every Candidate and branch has an accepted terminal outcome, every live Integration carries current qualification proof, deterministic and actual-Mac gates pass, no Partial remains, and the generated closeout report and exhaustive ledgers reconcile reproducibly to the immutable workbook.
